# 경계 설계: 어디를 기준으로 나눌 것인가

MFA를 처음 설계할 때 가장 많이 실수하는 지점은 **컴포넌트 단위 분해로 시작하는 것** 입니다.

대부분의 경우 안전한 시작점은:

- 페이지 기준
- 라우트 기준
- 비즈니스 플로우 기준

입니다.

## 좋은 분해 기준

### 1. 사용자 여정이 끊기는 지점인가

예:

- `/products`
- `/checkout`
- `/me`

이렇게 페이지 전환이 명확한 곳은 셀 경계로 잡기 쉽습니다.

이유:

- 실제 라우트 전환은 UI state를 자연스럽게 초기화시킵니다
- 앱 간 실시간 UI state 공유 필요가 줄어듭니다

### 2. 한 팀이 end-to-end로 소유할 수 있는가

다음이 한 팀의 책임 안에 들어갈 수 있어야 좋습니다.

- 요구사항
- UI
- API 조합
- 배포
- 장애 대응

### 3. 독립 배포 가치가 있는가

다음 질문을 해보면 됩니다.

- 이 영역은 자주 바뀌는가
- 다른 영역과 배포 주기가 다른가
- 분리했을 때 병목이 줄어드는가

독립 배포의 가치가 없다면, 굳이 쪼갤 이유가 약합니다.

### 4. 앱 간 실시간 협업이 너무 많이 필요한가

이 기준이 매우 중요합니다.

두 앱이 계속 아래를 주고받아야 한다면:

- 현재 선택값
- 입력값
- 탭 상태
- 중간 플로우 상태

그 둘은 사실 하나의 셀일 가능성이 큽니다.

좋은 MFA는 "앱 간 통신이 화려한 구조"가 아니라, **앱 간 통신이 많이 필요 없도록 경계를 잘 자른 구조** 입니다.

## KYC를 셀로 보면 어떤 의미인가

KYC를 셀로 본다는 것은:

- 큰 서비스 안의 기능 단위로 본다
- 상위 제품 경험 안에서 자연스럽게 이어진다
- 하지만 내부적으로는 작은 앱처럼 동작한다

즉:

> KYC는 서비스 안에 들어 있는 기능이지만,
> 자기 안에서 라우트, 화면, 상태, API, 흐름 제어를 스스로 책임지는 완결된 mini app 이다.

## `processes` 중심이라는 말의 실제 의미

이 표현은 표준 강제 구조는 아니지만, KYC 같은 흐름형 도메인을 설명할 때 매우 유용합니다.

예를 들어 일반적인 구조는 보통 이렇게 시작합니다.

```text
pages/
components/
store/
api/
```

하지만 KYC의 핵심은 단순 페이지 모음이 아니라 **흐름 제어** 입니다.

예:

- 촬영 실패 -> 재시도
- 외국인 사용자 -> 다른 단계
- 기존 가입자 -> 종료
- 성공 -> 다음 플로우 이동

이런 흐름은 `pages` 만으로 설명하기 어렵습니다.

그래서 `processes` 를 두고, "처음부터 끝까지 어떻게 흘러가야 하는지"를 모으는 식의 구조가 유효합니다.

예시:

```text
apps/
  kyc/
    src/
      app/
        main.ts
        router.ts
      processes/
        kyc-flow/
          orchestrator/
          strategies/
          usecases/
          model/
      pages/
      features/
      entities/
      shared/
```

여기서 핵심은:

- `pages` = 화면
- `features` = 기능 조각
- `entities` = 도메인 데이터
- `processes` = 흐름 제어와 orchestration

즉 KYC를 셀로 보면, **"화면 모음"이 아니라 "흐름을 가진 기능 앱"** 으로 보는 것이 맞습니다.

## KYC를 앱으로 보면 어떤 의미인가

반대로 KYC를 `app` 으로 본다는 것은:

- 큰 서비스 내부 기능이 아니라
- 거의 별도 시스템처럼 취급된다는 뜻입니다

느낌으로는:

- `cell` = 집 안의 방
- `app` = 옆 건물

즉:

- 자체 라우팅 가능
- 자체 배포 가능
- 자체 팀 ownership 가능
- 여러 서비스에서 공통 재사용 가능

## KYC를 앱으로 보는 것이 맞는 경우

### 1. 여러 서비스가 공통으로 사용한다

예:

- 신규 가입
- 카드 발급
- 송금
- 대출

이 모두가 KYC를 공통으로 쓰면, KYC는 특정 서비스의 페이지가 아니라 **공통 플랫폼 기능** 에 가까워집니다.

### 2. 독립 배포 가치가 매우 크다

예:

- 법규 대응으로 KYC만 자주 수정된다
- 외부 인증기관 연동만 별도 릴리즈가 잦다

이 경우 전체 제품과 같이 배포하는 비용이 커집니다.

### 3. 보안/격리 요구가 높다

예:

- 신분증
- 카메라
- 민감정보
- 외부 본인확인 연동

이런 처리 때문에 더 강한 격리와 독립 운용이 필요할 수 있습니다.

## 셀 vs 앱 판단 기준

### KYC가 셀인 경우

- 지금 서비스 안의 한 플로우로 이어져야 한다
- 같은 제품 경험 안에서 자연스럽게 동작해야 한다
- shell과 함께 움직이는 것이 더 자연스럽다

### KYC가 앱인 경우

- 여러 서비스에서 공통으로 쓴다
- 독립 배포 가치가 크다
- 독립 팀/독립 운영이 필요하다
- 플랫폼 기능에 가깝다

## 언제 MFA를 하지 않는 게 맞나

다음에 가깝다면 먼저 **modular monolith** 를 고려하는 편이 안전합니다.

- 팀이 아주 작다
- 독립 배포 가치가 약하다
- 경계가 아직 불명확하다
- 앱 간 상태를 자주 주고받아야 한다
- 운영 자동화가 약하다

MFA는 "미리 깔아두는 미래 대비책"으로는 비용이 큰 경우가 많습니다.

## 처음 도입할 때 안전한 설계

1. 한 프레임워크로 먼저 시작한다
2. 라우트 기준으로 2~3개 셀만 먼저 분리한다
3. shell은 얇게 유지한다
4. 앱 간 계약은 URL을 우선 사용한다
5. 공통 모듈은 정말 필요한 것만 뽑는다
6. 전역 store를 기본값으로 두지 않는다
7. 가능하면 같은 origin으로 먼저 배포해본다
8. shared dependency 전략은 하나로 통일한다

## 한 문장 정리

**많이 대화해야 하는 둘은 원래 하나여야 할 수 있다.**
