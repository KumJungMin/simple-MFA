# 운영과 거버넌스: 기술보다 어려운 부분

MFA의 진짜 비용은 종종 코드보다 운영과 거버넌스에서 더 크게 나타납니다.

왜냐하면 분리된다는 것은 곧 다음이 늘어난다는 뜻이기 때문입니다.

- repo
- 빌드/배포 파이프라인
- 런타임 경계
- 서버/자산 관리 포인트
- 계약 관리 포인트
- 품질 기준 조율 포인트

## 디자인 시스템과 공통 UI

MFA에서 "하나의 제품처럼 보이게 만드는 일"은 생각보다 어렵습니다.

그래서 보통 아래는 플랫폼 레이어로 올릴 가치가 있습니다.

- design tokens
- typography / spacing rules
- base button / input
- styleguide
- global error handling

하지만 여기에도 원칙이 있습니다.

**자주 같이 바뀌어야 하는 것만 공통화** 해야 합니다.

셀 특화 UI까지 모두 공통 컴포넌트로 흡수하기 시작하면, 이번에는 디자인 시스템이 새로운 모놀리스가 됩니다.

## 운영에서 꼭 필요한 것

### 1. 계약 문서화

아래는 모두 계약입니다.

- URL 규칙
- 이벤트 이름
- shared module public API
- auth contract

이 계약은 말로만 공유하면 쉽게 무너집니다.

### 2. 계약 테스트

대표적인 대상:

- route contract
- event payload schema
- shared module public API

즉, "연결점"을 테스트해야 합니다.

### 3. production-like 통합 테스트

각 앱을 standalone으로만 개발하면 편하지만, 실제 운영 조합 환경에서는 쉽게 깨질 수 있습니다.

그래서:

- 로컬 단독 개발
- 정기적인 통합 환경 테스트

를 함께 가져가는 것이 중요합니다.

### 4. 성능 관리

독립 번들은 장점이 있지만, 공통 의존성 중복으로 payload가 늘 수 있습니다.

특히:

- React
- Vue
- router
- design system

같은 대형 의존성의 공유 정책을 일찍 정해야 합니다.

## 언제 MFA를 하지 않는 게 맞나

다음에 가깝다면 먼저 modular monolith가 더 적절할 수 있습니다.

- 팀이 1개거나 아주 작다
- 독립 배포 가치가 약하다
- 경계가 아직 명확하지 않다
- 화면끼리 상태를 계속 주고받아야 한다
- 운영 자동화 수준이 낮다

MFA는 작은 제품에 "미리 가져다 놓는 미래 대비책"으로는 종종 너무 비쌉니다.

## 처음 도입하는 팀에게 추천하는 안전한 순서

1. 한 프레임워크로 시작한다
2. route 기준으로 2~3개 셀만 먼저 분리한다
3. shell은 얇게 유지한다
4. 앱 간 계약은 URL 우선으로 설계한다
5. 공통 모듈은 `auth`, `api`, `design-system`, `config` 정도만 먼저 만든다
6. global store는 기본값으로 두지 않는다
7. 가능하면 같은 origin으로 먼저 배포한다
8. shared dependency 전략은 하나만 고른다

## 실무적인 위험 신호

다음 징후가 보이면 구조를 다시 봐야 합니다.

- 앱 간 이벤트가 계속 늘어난다
- 전역 store shape 논의가 자주 나온다
- 셀들이 서로의 내부 타입을 많이 안다
- shell이 각 셀의 비즈니스 규칙까지 안다
- 한 셀 수정에 여러 앱 동시 배포가 필요하다

이런 징후는 "경계가 잘못 잘렸거나, shared contract가 너무 두껍다"는 신호일 가능성이 높습니다.

## MFA를 운영할 때 꼭 지켜야 할 원칙

### 원칙 1. 계약은 좁게

- route contract
- shared module public API
- 최소한의 event contract

계약이 넓을수록 독립성은 빠르게 약해집니다.

### 원칙 2. 공유는 적게

공유가 많아질수록:

- 배포가 같이 움직이고
- 버전 충돌이 늘고
- 운영이 어려워집니다

### 원칙 3. shell은 얇게

shell이 비즈니스 로직을 많이 먹기 시작하면, 결국 셀들은 진짜 독립 단위가 아니라 shell의 하위 모듈처럼 변합니다.

### 원칙 4. 통합 환경을 자주 본다

standalone 개발은 편리하지만, 실제 사용자 경험은 통합 환경에서 결정됩니다.

## 마지막으로 기억할 문장

**MFA의 성공 조건은 "얼마나 잘 쪼갰는가"보다, "얼마나 적은 계약만으로도 함께 동작하게 만들었는가"에 가깝습니다.**
