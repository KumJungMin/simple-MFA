# 한 페이지에 여러 앱을 삽입하는 방식 이해하기

이 문서는 **하나의 페이지 안에 여러 개의 프론트엔드 앱을 함께 렌더링하는 방식**을 처음 보는 사람도 이해할 수 있게 설명하는 가이드입니다.

특히 아래 같은 상황을 머릿속에 두고 읽으면 좋습니다.

- 메인 앱은 React다
- 이벤트 배너 영역은 별도 팀이 만든 앱이다
- 추천 상품 위젯은 독립적으로 배포된다
- 페이지 전체를 갈아엎지 않고 일부 영역만 교체하고 싶다

이런 구조는 마이크로 프론트엔드에서 자주 등장합니다. 다만 중요한 점은, 이것이 "무조건 좋은 구조"라서 쓰는 것이 아니라 **영역 단위 독립성**이 필요할 때 선택하는 방식이라는 점입니다.

## 0. 먼저 한 문장으로 이해하기

**한 페이지에 여러 앱을 삽입한다는 것은, 페이지 전체는 하나의 셸 또는 메인 앱이 관리하되, 특정 영역만 별도의 독립 앱이 렌더링하도록 조합하는 구조**입니다.

쉽게 말하면:

- 페이지 전체를 책임지는 상위 앱이 있고
- 그 안의 일부 영역은 다른 앱이 들어와서 그린다

라고 보면 됩니다.

## 1. 우리가 익숙한 구조와 무엇이 다른가

보통은 이렇게 생각하기 쉽습니다.

```text
[하나의 페이지]
   └─ [하나의 프론트엔드 앱]
```

예를 들어 하나의 React 앱이:

- 헤더
- 메인 콘텐츠
- 배너
- 공지
- 추천 상품

을 전부 렌더링하는 구조입니다.

그런데 실무에서는 꼭 이렇게만 가지 않습니다.

```text
[하나의 페이지]
   ├─ [메인 앱]
   ├─ [이벤트 배너 앱]
   └─ [이벤트 리스트 앱]
```

즉, **하나의 HTML 문서 안에서 여러 개의 앱이 동시에 살아 있는 구조**도 가능합니다.

이때 각 앱은:

- 별도 코드베이스를 가질 수도 있고
- 별도 배포 파이프라인을 가질 수도 있고
- 서로 다른 프레임워크를 사용할 수도 있습니다

## 2. 왜 이런 구조를 쓰는가

이 방식은 단순히 "기술적으로 가능해서" 쓰는 게 아닙니다. 보통 아래 이유 중 하나 이상이 있어야 가치가 생깁니다.

### 2-1. 특정 영역만 독립적으로 배포하고 싶을 때

예를 들어 이벤트 배너만 자주 바뀐다면:

- 메인 앱 전체를 재배포하지 않고
- 이벤트 배너 앱만 수정해서 배포

하고 싶을 수 있습니다.

이 경우 영역 단위 독립 배포의 가치가 생깁니다.

### 2-2. 팀 경계를 기능 영역에 맞추고 싶을 때

예를 들어:

- 메인 앱 팀
- 이벤트 팀
- 추천 상품 팀

이 분리되어 있다면, 페이지 일부를 별도 앱으로 떼어 두는 편이 ownership을 나누기 좋습니다.

### 2-3. 프레임워크가 달라도 붙이고 싶을 때

예를 들어:

- 메인 앱은 React
- 이벤트 위젯은 Vue
- 공지 위젯은 바닐라 JS 또는 Web Components

처럼 섞일 수 있습니다.

즉, **페이지를 하나의 앱으로만 보지 않고, 영역 단위로 조합하는 것**이 핵심입니다.

## 3. 이 구조를 이해할 때 꼭 분리해서 봐야 하는 3가지

처음 볼 때 가장 많이 헷갈리는 이유는, "로드", "렌더", "페이지 소유"가 한꺼번에 섞여 보이기 때문입니다.

아래 세 가지를 분리해서 보면 이해가 훨씬 쉬워집니다.

### 3-1. 누가 페이지를 소유하는가

예:

- 메인 셸이 `/home` 페이지를 소유
- 그 안의 일부 영역에 위젯 앱을 삽입

이건 **페이지 ownership**입니다.

### 3-2. 앱 코드를 어떻게 불러오는가

예:

- 이미 번들에 포함되어 있음
- 런타임에 remote 스크립트로 로드
- iframe으로 별도 문서를 불러옴

이건 **코드 로딩 방식**입니다.

### 3-3. 불러온 앱을 어떤 방식으로 DOM에 붙이는가

예:

- `iframe`
- `custom element`
- `mount(container)` 함수 호출

이건 **마운트 방식**입니다.

이 셋을 섞지 말고 보면 좋습니다.

예를 들어 이 저장소의 KYC 페이지는:

- 페이지 ownership: `shell-react`
- 코드 로딩: remote 스크립트 동적 로드
- DOM 마운트: custom element

라는 3단 구조로 이해할 수 있습니다.

## 4. 대표적인 삽입 방식

한 페이지에 여러 앱을 넣는 대표 방식은 아래와 같습니다.

### 4-1. iframe

페이지 안에 또 다른 페이지를 넣는 방식입니다.

```html
<iframe src="https://event.example.com"></iframe>
```

장점:

- 격리가 강합니다
- CSS/JS 전역 충돌이 적습니다
- 외부 시스템을 넣기 쉽습니다

주의할 점:

- 높이 조절, 포커스, 히스토리 연동이 번거롭습니다
- UX가 분리되기 쉽습니다
- cross-origin이면 `postMessage()` 같은 별도 통신이 필요합니다

보통은 **외부 시스템 임베드**, **강한 격리**, **레거시 임시 통합**에 더 잘 맞습니다.

### 4-2. custom element

브라우저의 커스텀 태그를 이용해 특정 앱을 삽입하는 방식입니다.

```html
<event-banner channel="A"></event-banner>
```

장점:

- 페이지 안에 자연스럽게 섞일 수 있습니다
- 프레임워크가 달라도 경계를 만들기 쉽습니다
- 위젯처럼 쓰기 좋습니다

주의할 점:

- 태그 이름, 속성, 이벤트 계약을 설계해야 합니다
- 스타일 격리 전략이 필요합니다
- 복잡한 상태를 억지로 속성으로 넘기면 결합도가 올라갑니다

### 4-3. 런타임 모듈 로딩 + mount 함수

특정 번들을 동적으로 불러와서 페이지 일부에 붙이는 방식입니다.

예를 들면:

```ts
const remote = await importRemoteSomehow();
remote.mount(container, { channel: 'A' });
```

이 방식은 custom element보다 조금 더 저수준이지만, lifecycle을 셸이 명시적으로 제어하기 좋습니다.

### 4-4. 중요한 구분: custom element와 런타임 로딩은 같은 말이 아니다

처음 보면 이 둘이 같은 것처럼 들릴 수 있지만, 실제로는 다릅니다.

- 런타임 로딩
  - 앱 코드를 **어떻게 불러오느냐**
- custom element
  - 불러온 앱을 **브라우저 DOM에 어떤 형태로 붙이느냐**

즉, 실무에서는 **런타임으로 스크립트를 로드한 뒤 custom element를 등록해서 붙이는 조합**이 자주 나옵니다.

이 저장소의 현재 구현도 이 방식입니다.

## 5. custom element 방식이란 무엇인가

쉽게 말하면 브라우저에게 **"이제부터 이런 태그도 이해해"** 라고 알려주는 것입니다.

예:

```html
<event-banner></event-banner>
```

원래 HTML에는 `event-banner`라는 태그가 없습니다. 하지만 우리가 직접 만들 수 있습니다.

즉, 이 태그는:

- 그냥 모양만 새로운 HTML 태그가 아니라
- 그 안에서 앱을 렌더링하는 입구

라고 이해하면 됩니다.

MDN 기준으로 custom element 이름은 **하이픈(`-`)을 포함해야** 하고, 페이지 전체에서 **전역적으로 고유한 이름**이어야 합니다. 그래서 `event-banner`, `mfe-kyc-app` 같은 형태가 흔합니다.

## 6. 가장 단순한 구조 그림

예를 들어 메인 앱 중간에 이벤트 배너 앱을 넣는다고 하면 이렇게 볼 수 있습니다.

```text
[메인 앱]
 ├─ 헤더
 ├─ 메인 콘텐츠
 ├─ <event-banner>
 └─ 푸터
```

여기서 `<event-banner>`는 **독립 앱이 들어올 마운트 지점**입니다.

메인 앱이 배너 UI를 직접 그리는 게 아니라, 이 위치를 하나 내어주는 것입니다.

## 7. 아주 단순한 동작 예시

메인 페이지는 이렇게 쓸 수 있습니다.

```html
<h1>메인 페이지</h1>

<section>
  <p>여기는 메인 앱이 관리하는 콘텐츠</p>
</section>

<event-banner channel="A"></event-banner>
```

중요한 부분은 이 줄입니다.

```html
<event-banner channel="A"></event-banner>
```

이 태그는 대략 이런 의미를 가질 수 있습니다.

- 이 위치에 이벤트 배너 앱을 렌더링해라
- `channel=A` 기준으로 동작해라

## 8. 이벤트 앱 쪽에서는 무엇을 하나

위젯 앱은 브라우저에 custom element를 등록합니다.

가장 단순한 예시는 이렇습니다.

```js
class EventBannerElement extends HTMLElement {
  connectedCallback() {
    this.innerHTML = `<div>이벤트 배너 영역</div>`;
  }
}

customElements.define('event-banner', EventBannerElement);
```

이 코드는 이렇게 읽으면 됩니다.

1. `EventBannerElement`라는 새 HTML 요소 타입을 만든다
2. 그 요소가 문서에 연결될 때 `connectedCallback()`이 호출된다
3. 그 안에 내용을 렌더링한다
4. 이 요소를 `event-banner`라는 태그 이름으로 등록한다

브라우저는 `<event-banner>`를 만나면:

- 이 태그가 등록되어 있는지 확인하고
- 등록된 클래스에 따라 동작시킵니다

참고로 초기화는 보통 `constructor`보다 `connectedCallback()`에서 처리하는 편이 안전합니다. 요소가 실제 DOM에 붙은 뒤 실행되기 때문입니다. 반대로 정리(cleanup)는 `disconnectedCallback()`에서 처리할 수 있습니다.

## 9. React나 Vue 앱도 넣을 수 있을까

가능합니다.

위 예시는 `innerHTML`만 넣은 아주 단순한 버전이고, 실무에서는 custom element 내부에서 React나 Vue를 마운트하는 형태가 많습니다.

예를 들어 React 개념 예시는 이렇습니다.

```ts
import React from 'react';
import { createRoot } from 'react-dom/client';

function EventBanner(props: { channel: string | null }) {
  return <div>이벤트 배너 - channel: {props.channel}</div>;
}

class EventBannerElement extends HTMLElement {
  connectedCallback() {
    const container = document.createElement('div');
    this.appendChild(container);

    const channel = this.getAttribute('channel');
    const root = createRoot(container);

    root.render(<EventBanner channel={channel} />);
  }
}

customElements.define('event-banner', EventBannerElement);
```

이 흐름은 다음과 같습니다.

1. 메인 페이지가 `<event-banner channel="A">`를 렌더링한다
2. 브라우저가 custom element를 인식한다
3. 등록된 클래스가 실행된다
4. 그 안에서 React root를 만들어 컴포넌트를 렌더링한다
5. 결과적으로 페이지 일부에 React UI가 꽂힌다

Vue도 같은 방식으로 가능합니다. 이 저장소의 KYC 앱이 실제로 그 예입니다.

## 10. 이 저장소에서는 어떻게 하고 있나

현재 저장소는 "한 페이지에 독립 앱을 삽입하는 방식"을 이미 보여주고 있습니다. 다만 예제가 이벤트 위젯이 아니라 KYC 도메인 앱이라는 점만 다릅니다.

현재 흐름은 이렇습니다.

1. 셸 페이지가 먼저 렌더링된다
2. 셸이 remote entry 스크립트를 동적으로 로드한다
3. remote 스크립트가 `mfe-kyc-app`을 custom element로 등록한다
4. 셸이 `<mfe-kyc-app token="..." api-base="/api">`를 렌더링한다
5. Vue KYC 앱이 그 안에서 실제 UI를 렌더링한다

관련 파일:

- [`apps/shell-react/src/KycPage.tsx`](../../apps/shell-react/src/KycPage.tsx)
- [`apps/shell-react/src/lib/load-remote.ts`](../../apps/shell-react/src/lib/load-remote.ts)
- [`apps/kyc-vue/src/remote.ts`](../../apps/kyc-vue/src/remote.ts)
- [`apps/kyc-vue/src/KycRemote.ce.vue`](../../apps/kyc-vue/src/KycRemote.ce.vue)

즉, 현재 저장소는:

- **페이지는 셸이 소유**
- **코드는 런타임에 로드**
- **DOM 삽입은 custom element 방식**

으로 동작합니다.

## 11. 폴더 구조를 어떻게 보면 좋은가

처음 보는 사람이 이해하기 쉬운 최소 구조는 아래처럼 볼 수 있습니다.

```text
apps/
  shell-app/
    src/
      pages/
      components/
  event-widget/
    src/
      bootstrap.ts
      EventBanner.tsx
      EventList.tsx
packages/
  shared-contracts/
```

의미:

- `shell-app`
  - 메인 페이지를 관리하는 앱
- `event-widget`
  - 이벤트 배너나 이벤트 리스트를 그리는 독립 앱
- `shared-contracts`
  - `Channel`, API 응답 타입, 이벤트 payload 타입 같은 계약만 공유

## 12. 메인 앱은 어디까지 책임져야 하나

여기서 핵심은 메인 앱이 **너무 많은 걸 알지 않는 것**입니다.

메인 앱은 보통:

- 어디에 넣을지
- 언제 보여줄지
- 간단한 설정값을 무엇으로 줄지

정도만 책임지는 게 좋습니다.

예:

```html
<event-banner channel="A"></event-banner>
```

즉 메인 앱은 컨테이너 역할을 하고, 위젯 앱의 내부 상세 로직까지 알지 않는 것이 좋습니다.

## 13. 데이터를 어떻게 넘길 것인가

이 부분이 가장 실무적입니다. "붙이는 것"보다 **어떤 데이터를 어떻게 주고받을지**가 구조를 더 많이 좌우합니다.

### 13-1. 단순한 설정값은 attribute로 넘겨도 된다

예:

```html
<event-banner channel="A" theme="dark"></event-banner>
```

이런 값들은 비교적 안전합니다.

- 문자열
- 숫자
- 간단한 플래그
- 렌더링 옵션

현재 저장소도 이 패턴을 씁니다.

```tsx
<mfe-kyc-app token={runtime.token} api-base={runtime.apiBase} />
```

### 13-2. 복잡한 비즈니스 객체는 속성으로 넘기지 않는 편이 좋다

예를 들어 이런 값은 조심해야 합니다.

- 전체 로그인 사용자 객체
- 권한 집합 전체
- 대형 cart 객체
- 실시간 UI 상태

이걸 메인 앱이 계속 속성으로 주입하기 시작하면:

- 위젯 앱이 메인 앱 구조에 의존하게 되고
- 독립 재사용성이 줄어들고
- 디버깅이 어려워지고
- 런타임 결합이 강해집니다

특히 React 18 같은 환경에서는 custom element에 전달한 값이 결국 **DOM attribute**로 내려가는 경우가 많기 때문에, 복잡한 객체 전달을 기본 패턴으로 삼지 않는 편이 좋습니다.

### 13-3. 그럼 위젯 앱은 데이터를 어디서 가져와야 하나

독립성이 중요하다면 보통 아래 순서를 먼저 고려합니다.

1. 위젯 앱이 직접 API를 조회
2. 또는 shared API utility를 통해 조회
3. 또는 BFF가 위젯에 맞는 데이터를 모아서 제공

즉:

```text
메인 앱 -> 위젯에 거대한 상태 주입
```

보다는

```text
위젯 -> 서버/API/BFF 기준으로 자기 데이터 조회
```

가 보통 더 안정적입니다.

다만 이것을 "무조건 항상 직접 조회가 정답"으로 이해하면 안 됩니다. 같은 페이지에 위젯이 아주 많아서 중복 요청이 과도해진다면:

- BFF로 응답을 합치거나
- shared API cache를 두거나
- 셸이 아주 얇은 초기 컨텍스트만 넘기는 식으로 조절할 수 있습니다

핵심은 **메인 앱이 위젯 내부 비즈니스 상태의 정본이 되지 않게 하는 것**입니다.

## 14. 이벤트 앱이 메인 앱에 알림을 보내고 싶다면

예를 들어 이벤트 배너를 클릭했을 때:

- 메인 앱 라우터 이동
- 분석 로그 기록
- 모달 열기

같은 처리가 필요할 수 있습니다.

이 경우 `CustomEvent`를 쓸 수 있습니다.

```ts
this.dispatchEvent(
  new CustomEvent('event-banner:click', {
    detail: { eventId: 'E001' },
    bubbles: true,
    composed: true
  })
);
```

이 예시의 의미:

- `detail`
  - 이벤트와 함께 전달할 데이터
- `bubbles: true`
  - 상위 DOM으로 이벤트가 버블링되게 함
- `composed: true`
  - Shadow DOM 경계를 넘어 바깥까지 전달되게 함

`composed: true`는 특히 web component가 Shadow DOM을 쓸 수 있기 때문에 알아두면 좋습니다. 이 값이 없으면 셸 쪽에서 이벤트를 못 받을 수 있습니다.

메인 앱은 native 이벤트 리스너로 받을 수 있습니다.

```ts
document.addEventListener('event-banner:click', (event) => {
  const customEvent = event as CustomEvent<{ eventId: string }>;
  console.log('이벤트 클릭:', customEvent.detail.eventId);
});
```

React 호스트에서는 버전과 구현에 따라 custom event 핸들링이 다르게 느껴질 수 있으므로, **`ref + addEventListener()` 또는 문서 루트에 native listener를 두는 방식**이 가장 안전한 출발점입니다.

## 15. lifecycle도 꼭 생각해야 한다

한 페이지에 삽입되는 위젯 앱은 보통 "잠깐 나타났다 사라질 수 있는 앱"입니다. 그래서 cleanup이 중요합니다.

예:

- 이벤트 리스너 해제
- 타이머 정리
- subscription 해제
- React/Vue root unmount

custom element를 직접 구현한다면 `disconnectedCallback()`에서 cleanup을 해줘야 합니다.

즉, custom element는 "태그만 예쁘게 만드는 기능"이 아니라, **작은 앱의 시작과 종료를 담는 lifecycle 컨테이너**라고 이해하면 좋습니다.

## 16. 스타일 격리는 자동으로 해결되지 않는다

많이 오해하는 부분입니다.

custom element를 쓴다고 해서 스타일 충돌이 자동으로 완전히 사라지는 것은 아닙니다.

고민할 지점:

- 전역 CSS reset이 서로 충돌하는가
- class name이 섞이는가
- Shadow DOM을 사용할 것인가
- 디자인 시스템 토큰은 공유할 것인가

예를 들어 Vue의 `defineCustomElement()`는 web component를 만들 때 유용하지만, 실제 스타일 격리 전략은 프로젝트가 추가로 정해야 합니다.

즉, custom element는 **경계 형성에는 도움**이 되지만, **스타일 거버넌스를 대신해주지는 않습니다.**

## 17. 어떤 경우에 잘 맞는가

이 방식은 보통 아래 같은 UI에 잘 맞습니다.

- 이벤트 배너
- 이벤트 리스트
- 추천 상품 위젯
- 공지 위젯
- 리뷰 요약 위젯
- 독립적으로 자주 바뀌는 보조 UI 블록

즉, **페이지 일부에 들어가는 self-contained widget**에 잘 맞습니다.

single-spa 문맥으로 보면 이런 것은 route를 소유하는 앱보다는, 필요 시 화면 일부에 렌더링하는 `parcel`과 비슷한 사고방식으로 이해할 수 있습니다.

## 18. 어떤 경우에는 잘 안 맞는가

반대로 아래처럼 **순차 흐름 제어가 핵심인 기능**에는 신중해야 합니다.

- KYC 전체 플로우
- 결제 플로우
- 로그인 플로우
- 인증/가입 전체 플로우

이런 기능은 단순 위젯보다:

- 라우팅 제어
- 페이지 ownership
- 에러 복구
- 인증 컨텍스트

가 더 중요합니다.

물론 기술적으로는 삽입할 수 있습니다. 하지만 구조적으로는 **페이지 단위 앱**이나 **도메인 셀**이 더 자연스러운 경우가 많습니다.

현재 저장소의 KYC도 기술적으로는 삽입형 custom element를 쓰지만, 업무 성격 자체는 단순 위젯이 아니라 **무거운 도메인 플로우**에 가깝습니다. 그래서 이 예제를 그대로 "모든 위젯에 대한 기본 해답"으로 받아들이기보다, **같은 기술을 더 큰 도메인 앱에도 쓸 수 있다**는 예시로 보는 편이 정확합니다.

## 19. 성능이 걱정되면 어떻게 생각해야 하나

한 페이지에 위젯이 많아지면 자연스럽게 이런 걱정이 생깁니다.

- 각 위젯이 API를 따로 불러서 느리지 않을까
- 중복 요청이 생기지 않을까
- 메인 앱이 데이터를 모아서 넘기는 게 낫지 않을까

이 질문에 대한 안전한 사고 순서는 보통 이렇습니다.

### 19-1. 먼저 자율성 기준으로 본다

위젯이 다른 페이지나 다른 제품에도 붙을 가능성이 크다면, 자기 데이터를 스스로 가져오는 쪽이 유리합니다.

### 19-2. 그다음 중복 비용을 본다

동일한 사용자 정보나 이벤트 목록을 여러 위젯이 반복 조회한다면:

- shared API cache
- BFF 집계 응답
- edge cache

같은 방법을 고려할 수 있습니다.

### 19-3. 마지막으로 셸 전달을 본다

셸이 아주 얇은 초기 컨텍스트만 주는 것은 괜찮습니다.

예:

- `channel`
- `locale`
- `theme`
- `apiBase`

하지만 셸이:

- 복잡한 사용자 객체 전체
- 위젯 내부 비즈니스 상태
- 실시간 UI 상태

까지 다 넘기기 시작하면 위젯 독립성이 빠르게 떨어집니다.

즉, 성능 최적화가 필요하더라도 기본 원칙은 유지하는 편이 좋습니다.

## 20. 한 번 더 아주 쉽게 비유해보기

이 구조를 비유하면:

- 메인 앱은 집 전체
- 위젯 앱은 집 안에 설치한 독립 가구/가전

입니다.

집 전체 동선과 구조는 메인 앱이 관리하지만, 특정 가전은:

- 별도 제조사가 만들고
- 별도 업데이트를 하고
- 필요한 전기와 신호만 받아서
- 자기 역할을 수행합니다

바로 그 느낌에 가깝습니다.

## 21. 핵심 요약

### 한 페이지에 여러 앱을 넣는다는 뜻

- 페이지 일부를 독립 앱이 렌더링하는 구조입니다

### 대표적인 방식

- `iframe`
- `custom element`
- 런타임 모듈 로딩 + `mount()`

### 이번 문서에서 중심으로 설명한 방식

- `custom element`

### 잘 맞는 예

- 이벤트 배너
- 이벤트 리스트
- 추천 상품 위젯
- 공지 위젯

### 중요한 원칙

- 메인 앱은 "어디에 넣을지"를 주로 관리합니다
- 위젯 앱은 "자기 UI와 자기 데이터"를 책임집니다
- 단순 설정값만 속성으로 넘기는 편이 안전합니다
- 복잡한 상태는 셸이 들고 흔들기보다 API/BFF/공유 모듈을 먼저 고려합니다
- `CustomEvent`는 알림 용도로 유용하지만, 핵심 도메인 흐름의 유일한 수단이 되면 위험합니다

## 22. 마무리

처음 보면 "한 페이지에 여러 앱을 넣는다"는 말이 꽤 낯설 수 있습니다. 하지만 구조를 단순하게 보면 결국 이렇게 정리할 수 있습니다.

> 메인 앱이 페이지 전체를 잡고, 특정 영역만 독립 앱을 꽂아 넣는 구조

그리고 이벤트 배너, 이벤트 리스트, 추천 위젯 같은 기능은 이 구조와 특히 잘 맞습니다.

반대로 KYC, 결제, 로그인 같은 무거운 플로우는 **정말 페이지 일부 위젯이어야 하는지**, 아니면 **도메인 앱/페이지 ownership이 더 맞는지**를 한 번 더 고민해보는 것이 좋습니다.
